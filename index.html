Лекция про git
Что такое git?
Большинство из вас, наверное, слышали о Git, но если нет -                          Git является инструментом управления версиями.                          Это просто красивая обертка - в сущности Git позволяет отслеживать, какие изменения вы делаете в файл.                          Вы, вероятно, использовали инструмент контроля версий, или, может быть, даже изобрел тот, который работает для вас.

Если вы когда-либо работали на бумаге с помощью Word, вы, вероятно, была создана система управления verison, которая работает для вас, сохранив файл с различными тегами "Версия".                         Причина кто-то может сделать это, потому что они, как имеющие резервную копию всех своих прошлых изменений. Если кто-то хочет удалить абзац, вы сохраните копию старой версии на всякий случай                         И когда вы сотрудничать с кем-то, это становится все более сложной задачей. Отслеживание изменений и слияние содержимое, добавленное разными людьми отнимает много времени.                         Как вы можете себе представить, это представляет проблему в разработке программного обеспечения.                         Вполне вероятно, что несколько человек будут работать на том же самом файле иногда даже одновременно.                         Даже небольшие программные проекты могут иметь десятки файлов в нескольких папках.                         Кроме того, разработчики часто экспериментировать с функциями, и это никогда не должно влиять на "мастер" копию проекта

Git на помощь! По сути, Git может быть это, это может быть история всех изменений, внесенных в каждый файл в проекте.                          Но, к сожалению, вы должны понять, как она работает, чтобы эффективно его использовать.                          Большинство людей при введении мерзавцу, выбрасываются в использовании, и, как правило, не понимают, как это работает. Когда что-то вы никогда не сталкивались прежде, чем произойдет, вы можете не знать, что делать.                          Я знаю, что удаленные проекты и загрузить новую копию репо, потому что я случайно создал конфликт какой-то.                          С помощью этого разговора, я хотел бы представить, что я называю большие идеи Git, которые помогут вам понять, как работает Git.
git - это расширение для вашего workflow
Первое, что нужно понимать, что Git является дополнением к вашему текущему рабочий процесс. Вы можете работать так, как вы обычно работаете, и все, что вам нужно сделать, это использовать Git каждый раз, когда вы хотите сохранить ваш прогресс

Прежде чем говорить о Git, давайте поговорим о рабочем процессе.                                                           Это простой пример рабочего процесса. Вы редактировать файл и сохранить файл. SPC.                              После того, как вы достигнете того, что вы считаете, стабильное состояние, вы можете сохранить файл под другим именем, как резервное копирование, или сохранить его на Dropbox или же вы, как правило, внедрить систему контроля версий                              При использовании Git однако, вы не должны сохранить другую версию файла.                              Когда вы будете готовы, чтобы сохранить текущую версию файла в качестве резервной копии, вы делаете это в Git, принимая снимок текущего состояния.                              Вы можете взять "снимок" текущего состояния файла с помощью команды под названием Git фиксации.                              Что на самом деле происходит, что, когда вы можете создать "снимок", он записывает текущее состояние всех файлов в каталоге отслеживаются.

git commit

Итак, что же снимок выглядит?                             Давайте предположим, что снимок выглядит как этот круг здесь.                         Как я упоминал ранее, снимок содержит текущее состояние файла или каталога.                         Когда мерзавец создает снимок, он также создает ряд и придает тот номер к нему.                         Git использует алгоритм для вычисления этого числа и                         Этот номер является уникальным, весьма маловероятно, что два таких номера будут такими же,                         Каждый снимок / фиксации вы создаете, будет иметь свой собственный номер, прикрепленный к нему. И это число называется HASH. Вы можете, как правило, вы можете использовать первые 6 или 7 символов, чтобы однозначно идентифицировать снимок в вашем проекте.                         Допустим, вы создали это первый коммит, и теперь вы сделаете больше изменений в файл. и хотите сохранить их.                         При внесении изменений в файлы и сохранить их с помощью Git фиксации, Git создает новый снимок с новым HASH.                         Git также делает несколько других вещей.                         1) Это позволяет создать сообщение, которое идет вместе с этой фиксации. Коммит должен иметь сообщение!                         2) При создании новой фиксации создан, он сохраняет ссылку на родительский фиксации. Снимок может иметь несколько родителей - то есть вы можете взять изменения из двух фиксаций и объединить их в одно целое.                         3) Независимо от фиксации представляет текущее состояние проекта, Git хранит ссылку, что ток фиксации в головке этикетки                         В VCS документе слово, мы создали несколько версий, но мы не отслеживать, как мы получили к каждому государству. Git позволяет сделать это за счет хранения всей этой информации                         Вам может быть интересно, где же мерзавец магазин все это - мы вернемся к этому.                         Это, как вы можете себе представить, формирует график.
А зачем мне все это?
и как с любым графиком, вы можете выбрать, чтобы рассказать историю.                         В этом случае, мы хотим работать на особенности программного обеспечения, и во время работы над ней мы получаем новую идею.                         Вы можете вернуться к предыдущему состоянию проекта, и ответвление к работе над новой идеей - не влияет на вашу текущую работу.                         И, когда вы будете готовы, вы можете вытащить эти изменения обратно в основной отрасли                         Вы можете вернуться назад и пересмотреть отдельные изменения, сделанные в какой-либо фиксации.                         Вы даже можете отменить изменения, применить эти изменения в другое состояние вашего проекта.                         Кроме того, можно отказаться от ветку целиком, или переключаться между ветвями.                         Отрасли являются удивительными, и мы будем говорить более способными ветви чуть позже.                         Дело в том, что этот граф существует.                         Этот график является документация, как ваш проект пришел, чтобы быть, то есть история вашего проекта график, который вы можете пройти по собственному желанию.
Идея #1
git история это граф
Так что это большая идея номер 1, история Git представляет собой график, который вы можете пройти, что позволяет перейти к любому "спасенных" состоянии в проекте

Let's talk about our workflow again. To actually use git we need to add a couple of steps to the workflow we talked about earlier. We group changes before committing them. And we do this using the git add command. if we open a folder and try to "git add" a file, git will throw us an error. it doesn't know where to add the file to. We need to create a .git folder first and we do that using git init. You only need to do this once for a project, at the every beginning. If a project is already using git, then that .git folder exists and you can run any git command. the git add command, adds the content you specify to a "staging area" You can add multiple files at the same time to the staging area And then, when you call the git commit function, git takes everything that is in the staging area and pushes it into a new snapshot Think of the content in the staging area as the changes you would like your next snapshot to represent SPC Before you commit these changes to a new snapshot, you might want to review your changes. SPC git status is used to review your changes, SPC And to actually save this snapshot you can use git commit
